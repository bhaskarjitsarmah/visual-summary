Background

The Single-Player Problem

Every AI product that came before was fundamentally single-player: one human, one conversation, one session. Knowledge died when the chat ended. Two teammates using the same AI still worked in complete isolation from each other. Claude Tag is built to break every one of these constraints.

Core Capabilities

Four Superpowers of @Claude

Anthropic describes Claude Tag through four verbs: learns, takes initiative, works asynchronously, and is everywhere you work. Each represents a step beyond the capability ceiling of previous AI interfaces.

🧠
@Claude learns
Channel-Scoped Persistent Memory
Claude persistently builds context within specific Slack channels — accumulating vocabulary, team workflows, past decisions, and recurring patterns. If your engineering channel discusses "the hydration bug" once, Claude learns that context and can reference it in future threads. Admins can optionally grant cross-channel read access. Memory writes are always logged for audit.
"Each connected system makes every other one more useful because Claude can combine context across them."
⚡
@Claude takes initiative
Ambient Mode — Proactive Monitoring
In ambient mode (opt-in), Claude monitors channels without being explicitly tagged. It flags updates that match your team's standing interests, follows up on forgotten threads, alerts to relevant signals from connected systems, and posts weekly digests — acting like a team member who's always in the room, not just one you have to summon. Anthropic recommends starting without ambient until you understand failure modes.
"Anthropic recommends keeping ambient off until teams understand failure modes."
⏳
@Claude works asynchronously
Long-Horizon Autonomous Tasks
Claude can schedule tasks for itself, pursuing multi-stage projects over hours or days without needing constant supervision. When assigned a complex task, it breaks it into stages, works through connected tools, and delivers results in a visible Slack thread — all while the team focuses on other work. Every step of the checklist stays visible to the entire channel.
"Tasks can be delegated while teams focus elsewhere, with results delivered asynchronously."
🔗
@Claude is everywhere
Cross-System Context Synthesis
With the right connections granted, Claude integrates Slack threads, Google Drive documents, GitHub issues and PRs, data warehouse queries, Datadog logs, Linear tickets, and more into single, synthesized answers. No single tool in your stack can answer "why are we seeing these anomalies?" by drawing on all five systems simultaneously — Claude Tag can.
"Pulling a thread from Slack, a doc from Drive, a ticket from a tracker, and a query from a warehouse into one answer no single tool could provide."
What Claude Tag is NOT: An autonomous agent that acts without oversight. Claude Tag is a bounded system — what it can reach is defined by admin-controlled permissions, sandbox isolation limits outbound traffic, and every action is logged. The "initiative" in ambient mode still operates within the permission envelope the admin defines.
Security Architecture

Agent Identity: A New Permission Model

The most architecturally significant innovation. Rather than borrowing human credentials, Claude Tag operates as a first-class service account with its own identity — shifting the access-control question from "what can this user do?" to "what can this agent do in this compartment?" Click any channel below to explore its permission scope.

The Key Insight: "What it can reach depends on the channel you're in, not on who you are." Claude's legal channel identity cannot access engineering code, even if the same human admin manages both channels. This makes agent permissions auditable, revocable, and bounded — just like a human employee's access card.

Why Service Account Identity Matters
With user-delegation, an AI acts as you — it has your GitHub token, your Drive access, your email. If the AI makes a mistake, it looks like you made it. If it's compromised, the attacker has your full permissions. Service account identity separates AI actions from human actions cleanly: the PR was opened by "Claude App", not you.
Private Channel Isolation
Each private channel gets a distinct Claude identity. Memories from legal review never appear in product discussions. Tools granted to data insights are blocked from HR. "A shared channel can never become a side door into someone's private documents." This is architecturally enforced, not just policy-based.
Infrastructure

The Ephemeral Sandbox Architecture

Claude Tag runs in Anthropic-hosted ephemeral sandboxes — not on your infrastructure. Each Slack thread gets its own sandbox. The key security invariant: sandboxes hold no credentials. An Agent Proxy injects credentials at the network boundary when a tool call is made. Outbound traffic to any host the admin hasn't allowed is blocked by default.

Default-Deny Outbound: The sandbox blocks all outbound traffic by default. An admin must explicitly allowlist each external host. This means a compromised tool cannot exfiltrate data to an arbitrary endpoint — only pre-approved destinations are reachable. No credential is ever resident in the sandbox environment itself.
Context Management

Agent Skills: Progressive Context Loading

Agent Skills are organized folders of instructions, scripts, and resources that Claude loads dynamically using progressive disclosure — never dumping everything into context at once. This solves the context-window efficiency problem: Claude knows enough to recognize relevance, but only loads deep detail when needed.

Three-Tier Loading Architecture
Each skill loads in three tiers based on relevance. Click each tier to understand when and why it loads.
1
Tier 1 — Always Loaded (Metadata)
Skill name, one-line description, entry conditions
~50 tokens
What it contains: The skill's name, a brief description of what it does, and the trigger conditions under which the full skill content should load.

Why this tier exists: Claude needs to know what skills are available without loading all their content. A workspace with 40 skills would fill the context window before the user even sends a message. Metadata allows Claude to recognize relevance cheaply.

Example: skill: "github-pr-review" — Reviews GitHub PRs for bugs, style, and performance. Loads when a PR URL is mentioned.
2
Tier 2 — Loaded on Match (SKILL.md)
Full skill instructions, tool invocations, expected outputs
~2–8k tokens
What it contains: The complete SKILL.md file — detailed instructions for how Claude should behave when this skill is active, which tools to call, what output format to use, and any constraints or safety notes.

Why this tier exists: Loading every full skill document for every turn wastes tokens and injects irrelevant instructions. Loading on match ensures Claude has the right instructions exactly when needed, without bloating context for unrelated tasks.

Trigger: Claude detects a GitHub PR URL in the user's message → loads the full github-pr-review SKILL.md → begins following its detailed review protocol.
3
Tier 3 — Loaded on Demand (Supplementary Files)
Referenced templates, schemas, examples, lookup tables
variable
What it contains: Files referenced from SKILL.md but not pre-loaded — templates, code examples, domain-specific lookup tables, or large schemas that are only needed for specific sub-tasks.

Why this tier exists: A PR review skill might reference a 15-page security checklist that's only needed for security-sensitive code paths. Loading it for every PR review wastes tokens; loading it only when Claude encounters authentication code is efficient and targeted.

Key principle: Information is never all loaded at once. The harness fetches exactly the files the active skill instructs it to load for the specific task at hand.
Why This Architecture Matters: A workspace might have 30+ skills (GitHub, Linear, Datadog, Google Drive, Confluence, customer CRM, custom internal tools…). Naively loading all of them for every message would consume 200k+ tokens before the user writes a word. Progressive disclosure keeps Claude fast, cheap, and focused — it only knows what's relevant to the current task.
Infrastructure Architecture

Managed Agents: Brain × Session × Sandbox

Claude Tag is built on Anthropic's Managed Agents infrastructure, which decouples three independently scalable and restartable components. This design enables resilience, parallelism, and the enormous performance improvements that made Claude Tag viable at enterprise scale.

🧠
Brain
Claude + stateless harness
Stateless
The LLM + orchestration logic. Can be restarted at any time — it re-attaches to the session log to resume. Multiple brains can share one session. Powers intelligent reasoning and tool selection.
📋
Session
Append-only event log
Durable
The canonical source of truth. Every Slack message, tool call, result, and observation is written here. Stored outside Claude's context window, retrieved via getEvents(). Survives sandbox and brain restarts.
📦
Sandbox
Ephemeral execution environment
Ephemeral
"Cattle, not pets." Provisioned fresh for each Slack thread. Discards all state when idle. No credentials stored. Multiple sandboxes can connect to one brain simultaneously. P50 TTFT improved ~60%.
Scenario: Sandbox Crashes Mid-Task
Claude Tag is running a 3-hour data analysis. Halfway through, the sandbox crashes (hardware failure). With legacy architecture: the entire task fails, all progress lost. With Managed Agents: the session log is intact (durable, not in the sandbox). A new sandbox is provisioned. The brain reconnects via its session ID. It calls getEvents() to recover the full event history. The task resumes from the last completed step. No data lost, minimal delay.
Performance: What Decoupling Enables
By separating session state (durable, external) from the brain (stateless, fast) and sandbox (ephemeral), new sandboxes can cold-start much faster — they don't need to restore any state, just connect to the session and begin executing. P50 time-to-first-token dropped ~60%. P95 dropped over 90%. This made interactive Slack response times practical for the first time.
