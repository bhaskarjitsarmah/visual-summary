The Problem

The AI Governance Paradox

Financial regulators demand absolute, mathematically verifiable compliance guarantees. LLMs operate by sampling from probability distributions — fundamentally nondeterministic. These two facts are in direct contradiction. Current guardrail solutions don't resolve it — they just push the probability higher.

⚡ LLM Reality
🎲 Stochastic sampling — same input, different output
📊 Probabilistic confidence — "99.9% certain"
🌫️ Continuous vector space — no hard boundaries
🔓 Jailbreakable — distribution can be shifted
⚔️
📜 Regulatory Requirement
✅ Deterministic — same input always same verdict
🔒 100% certainty — no statistical hedging allowed
📐 Hard thresholds — capital limits are binary
🛡️ Provable — must demonstrate to regulators
The Knight Capital Lesson: On August 1, 2012, Knight Capital's algorithmic trading system deployed legacy code by accident. In 45 minutes — before any human could intervene — it lost $440 million. The probabilistic guardrails that existed were insufficient to catch the cascading errors deterministically. This single incident triggered the SEC's Market Access Rule (Rule 15c3-5) and demonstrates why "highly likely to be safe" is not the same as "provably safe."
Why Existing Guardrails Fail
• NVIDIA NeMo Guardrails — runs a secondary LLM to classify inputs. Probabilistic. ~500ms overhead. Can be bypassed by adversarial prompts.

• Guardrails AI — syntactic validators and regex. Fast but low logic depth. Cannot handle multi-variable financial constraints like "trade volume ≤ 10% of capital AND price deviation ≤ 2%".

Neither satisfies SEC Rule 15c3-5's requirement for "direct and exclusive control."
The Proposed Resolution
Separate the AI's thinking (probabilistic, allowed) from its acting (deterministic, required).

Every compliance rule is formalized into Lean 4 theorem-prover code once, offline. At runtime, every proposed agent action is submitted as a mathematical conjecture. The Lean kernel checks it — returning exactly True or False — in ~5 microseconds. No LLM involved in the verdict.
The Translator

The Aristotle Model

The hardest part of the system isn't the formal verification — it's translating human-written compliance policies from natural English into Lean 4 code. Aristotle (Harmonic AI) is a neural-symbolic model built for exactly this: auto-formalization at research-grade precision.

What makes Aristotle unusual: Most LLMs predict the next token. Aristotle uses the Lean 4 compiler as a live reward signal inside a reinforcement learning loop. Every generated proof attempt is checked by the Lean kernel — incorrect reasoning gets a compiler error with an exact counterexample. Aristotle ingests that error and tries again. This is why it achieved gold-medal-equivalent results on the 2025 International Mathematical Olympiad (5/6 problems), producing machine-verifiable Lean 4 solutions.

How the Error-Repair Loop Works
1. Aristotle's informal reasoning engine generates a first-draft Lean 4 translation

2. Lean compiler checks it — often returns a semantic error, e.g. "confused strictly decreasing with weakly decreasing"

3. Aristotle ingests the exact error and counterexample, repairs the logical gap

4. Repeat until the Lean kernel accepts the proof

The compiler is the teacher — Aristotle never has to be told what went wrong, only shown the formal error.
Epistemic Gaps and "Sorry" Tactics
When Aristotle cannot complete a sub-proof, it places a sorry placeholder — Lean's way of marking an incomplete proof. The system can then autonomously attempt to fill multiple sorry markers within a single theorem, bridging from high-level policy intent to granular logic gates.

This iterative gap-filling process is what makes it practical for real compliance documents, which often contain implicit assumptions humans don't write down explicitly.
The Verifier

Lean 4 — Mathematical Proof as Code

Lean 4 is a dependently typed formal programming language built on the Calculus of Inductive Constructions (CIC). It can express and verify mathematical theorems. Its key property for this application: proof generation (slow, one-time) is completely decoupled from proof checking (fast, every query).

The Two-Speed Architecture
Proof generation — translating a compliance rule into Lean 4 — can take seconds or minutes. It runs asynchronously, offline, during the policy configuration phase. Done once per rule.

Proof checking — verifying that a specific action satisfies the pre-compiled axioms — runs synchronously at query time. The Lean trusted kernel performs this in ~5 microseconds. This is what executes 1,000× faster than NeMo Guardrails.
Why Lean 4 Is So Fast at Checking
The Lean 4 kernel is a highly optimized C++ implementation performing type-checking against pre-compiled binaries. At runtime, it doesn't re-prove anything from scratch — it simply verifies that the proposed action's parameters satisfy the algebraic constraints already encoded.

It's similar to how a compiler's type-checker can verify type safety in microseconds without "re-understanding" the whole program — it just checks structural compatibility.
Latency Comparison — Guardrail Solutions (log scale)
The AWS Cedar Benchmark: Amazon's Cedar authorization language was formally verified using Lean 4, and the same differential testing harness was used to benchmark the Lean-Agent protocol. Average verification time: 5 microseconds per input — faster than equivalent Rust code (7 μs). This is the empirical baseline the paper uses to claim production-grade performance.

The System

The Two-Phase Architecture

The Lean-Agent Protocol separates compliance into two phases with fundamentally different timing and computational characteristics. Phase 1 is slow, offline, and done once. Phase 2 is fast, synchronous, and runs on every agent action.

Phase 1 — Async (One-Time)
Policy Configuration
1
Compliance officer writes policy in plain English: "No single trade may exceed 10% of available capital."
2
Aristotle auto-formalizes the policy into a Lean 4 axiom, iteratively repairing semantic errors via compiler feedback.
3
Lean kernel compiles the axioms and theorems. Aristotle patches any remaining logical gaps.
4
Policy Environment stored — an immutable set of compiled Lean 4 axioms, theorems, and definitions. Ready for runtime.
Phase 2 — Sync (Every Action)
Runtime Authorization Loop
1
Agent generates intent: execute_trade(AAPL, 50000, market)
2
Orchestrator intercepts — tokenizes the request, extracts parameters, captures current capital and market state.
3
Conjecture formulated — maps the action to a formal mathematical statement against the Policy Environment axioms.
4
Lean kernel verifies in ~5 μs — returns exactly: True → execute or False → block.
The Critical Separation: The AI agent is allowed to reason probabilistically — that's unavoidable and fine. But its actions are unconditionally gated by a deterministic mathematical check that the AI cannot influence. No jailbreak, adversarial prompt, or model drift can bypass the Lean kernel — it doesn't speak natural language. It only speaks formal logic.

Critical View

What the Paper Leaves Open

The Lean-Agent Protocol is a compelling architecture, but several real challenges aren't fully resolved in the paper. Here's an honest assessment of what would need to be true for this to work in production.

📝
Preprint by Independent and High School Researchers
The paper is a preprint (not peer-reviewed). The authors are Devakh Rashie (independent researcher) and Veda Rashi (Thomas Jefferson High School). The core technical synthesis is sound and the citations are legitimate, but the claims have not been reviewed by domain experts in financial regulation or formal methods. Treat it as a strong proposal, not a deployed system.
High Caveat
⚙️
Translation Layer Is the Weakest Link
The Lean kernel is provably correct — but Aristotle's translation from natural language to Lean code is still probabilistic. If Aristotle mistranslates a policy (e.g., translates "strictly less than" as "less than or equal to"), the compiled axiom is formally correct but semantically wrong. The paper acknowledges this and proposes MenTaL constraints, but the translation layer remains a non-trivial attack surface.
Medium-High Severity
📚
Lean 4 Expertise Is Scarce in Finance
Adopting this system requires compliance officers who can review Lean 4 code (or trust Aristotle entirely), engineers who can maintain the Policy Environment, and auditors who can interpret formal proof traces. None of these skills exist at scale in the financial industry. The paper provides a three-phase adoption roadmap but doesn't minimize the change management challenge.
Medium Severity
🔀
Conflicting Regulations Not Addressed
SEC, FINRA, OCC, CFPB, and EU AI Act sometimes have overlapping or contradictory requirements across jurisdictions. The paper treats each regulation independently. What happens when two compiled axioms in the Policy Environment are logically contradictory? The paper doesn't address conflict resolution between axioms — a non-trivial problem in dependently typed systems.
Medium Severity
📈
Genuine Technical Contribution
Despite the above, the core architecture is technically sound and novel. Decoupling proof generation from proof checking is a well-established principle in formal methods — the paper's contribution is applying it to the specific problem of real-time financial compliance. The 5 μs AWS Cedar benchmark is real. The Aristotle IMO results are independently verified. The regulatory analysis is careful and accurate.
