Sakana Fugu: When
Orchestration Beats Scale

Background

The New Scaling Axis

Traditional AI scaling has two axes: pre-training compute and parameter count. Sakana AI proposes a third: orchestration quality. Rather than training a bigger model, train a smarter coordinator over a pool of specialized workers.

Three Ways to Scale AI Capability
📊
Pre-Training Scale
Train on more tokens, more compute. GPT-4 → GPT-5.5. Costs billions, takes months, requires access to massive data.
⚙️
Fine-Tuning / RLVR
Distill capability, GRPO training. Domain-specific improvements. Requires parameter access — impossible for closed APIs like GPT-5.5.
🐡
Orchestration Learning
Train a coordinator to route tasks to the right specialist. Works at the API level — no weights needed, works with any closed-source model.
Fugu's Claim: Orchestration ≈ +1 Model Generation of Capability
The key insight: no single frontier model dominates all domains. GPT-5.5 leads on coding tasks; Gemini on scientific reasoning; Claude on analysis. A coordinator that correctly routes each query captures the best of all three simultaneously.
Why This Matters: After the US Department of Commerce imposed export controls on Anthropic's Fable 5 and Mythos models (June 12, 2026), Sakana positioned Fugu as frontier capability without export restriction — reaching comparable performance through orchestration of already-accessible models. This framing is contested (see Critical View section), but the underlying technical approach is independent of geopolitics.

Fugu — TRINITY Architecture

Evolutionary Coordination

TRINITY assigns Thinker / Worker / Verifier roles to frontier models, turn-by-turn, using a 0.6B backbone and a tiny 10K-parameter selection head trained by evolutionary search — not gradient descent. The choice to use evolution instead of RL is a deliberate response to the sparse, high-variance reward landscape of multi-turn tasks.

Why Evolution, Not RL? Three reasons: (1) Rewards are sparse binary signals — success or failure only at the very end of a multi-turn chain. (2) Parameter coupling is weak — the selection head is nearly independent across layers, so gradient methods suffer high variance. (3) SFT labels for multi-turn routing are theoretically prohibitive: 8.7×10¹⁰ queries to exhaustively sample. Sep-CMA-ES (separable CMA-ES) achieves 0.615 on LiveCodeBench vs. 0.253 for REINFORCE — a 2.4× advantage.
The Three Roles
• Thinker — generates the plan, reasons about the problem, structures the approach. Critical for hard reasoning tasks: removing Thinker costs -1.75pp average accuracy.

• Worker — implements the plan, executes code, retrieves information.

• Verifier — checks the worker's output, catches errors, validates correctness. The quality-control layer.
The Penultimate Token Insight
The backbone reads the hidden state h ∈ ℝᵈ at the penultimate token position — not the final token. This is non-obvious: the final token intuitively seems most informative, but it actually carries less distinguishing signal for routing decisions.

Using the final token instead drops performance by -5.78 percentage points. The penultimate position captures more context about the task type before the model "commits" to an output.

Fugu-Ultra — The Conductor

GRPO-Trained Workflow Generation

The Conductor is a 7B model (Qwen2.5-7B, fine-tuned with GRPO) that generates full natural-language agentic workflows — specifying which models handle which subtasks, in what order, and with what visibility into each other's outputs. It doesn't just route: it architects the entire multi-agent strategy.

What the Conductor Outputs: A structured Python-list workflow with three components:
(1) Model IDs — which frontier model handles each step
(2) Natural-language subtasks — specific instructions for each agent
(3) Access lists — which agent can see which prior outputs (defines communication topology)

The access lists are the key innovation: they enable emergent debate, verification loops, and recursive delegation — all without being explicitly programmed.

GRPO Training Details
• Base model: Qwen2.5-7B
• 200 training iterations
• Batch size: 256, 64 rollouts per question
• Training data: 960 problems across MATH, MMLU, RLPR, LiveCodeBench
• Reward: binary task success (did the final answer match?)
• Larger (7B) vs. smaller (3B) Conductor: gap comes from better subtask prompt engineering, not different model selection
Dual Memory Architecture
Intra-workflow isolation: Agents only see outputs explicitly listed in their access list — preventing "orchestration collapse" where agents copy each other's errors and converge on a shared wrong answer.

Persistent shared memory: Agents retain environment interaction history across separate workflows — enabling multi-session learning and continuity within long tasks.

Notably, plain Fugu beats Fugu-Ultra on SciCode (60.1 vs 58.7), τ³ Banking (21.7 vs 20.6), and Long Context Reasoning (74.7 vs 73.3) — the speed-quality tradeoff is real, and Fugu-Ultra's additional orchestration overhead doesn't always pay off for specialized or latency-sensitive tasks.

The Specialization Insight: Frontier models aren't uniformly capable — GPT-5.5 leads on Terminal Bench and MRCRv2 (algorithmic/code), Gemini leads on GPQA Diamond and scientific tasks, Claude leads on SWE-Bench Pro and complex analysis. Fugu learns to exploit these specializations without needing to be told they exist. The routing emerges from the reward signal alone.
Why Not Just Always Use the Best Model?
No model is best on everything. Even GPT-5.5 scores only 85.3% on LiveCodeBench vs. Fugu-Ultra's 93.2%. The best-of-all-tasks winner requires knowing which model is best for this specific query — a meta-skill that Fugu learns and individual models don't have.
Zero-Shot Transfer
TRINITY's coordinator transfers across tasks it wasn't trained on. Average zero-shot performance: 54.21% across AIME, BigCodeBench, MT-Bench, GPQA-Diamond — without any task-specific training. The selection head learns a general routing heuristic, not task-specific memorization.
Critical Perspective

The Honest Assessment

Sakana Fugu is technically interesting, but several contested claims and practical concerns deserve careful scrutiny. Here are the five most important challenges to the paper's framing.

🎯
Benchmark Credibility Gap
Fable 5 and Mythos are export-controlled and unavailable, so comparisons rely on vendor-reported scores, not head-to-head testing. Independent estimation puts Fable 5 SWE-Bench Pro at ~86.0% vs. Fugu-Ultra's 73.7% — a significant gap the paper's framing obscures. Co-founder David Ha later walked back the strongest "parity" claims.
High Severity
🔒
Opaque Black-Box Routing
Users cannot see which frontier model answered their query. This creates auditability problems for regulated domains (healthcare, legal, finance). If a medical answer came from Model A rather than the more reliable Model B for that task type, neither the user nor the operator has visibility. ~60% of billed tokens reportedly go to invisible back-channel orchestration, further obscuring the cost structure.
Medium-High Severity
🗺️
Overstated AI Sovereignty Narrative
Fugu's worker pool includes GPT-5.5 (OpenAI) and Claude Opus 4.8 (Anthropic) — American models. Fugu redistributes rather than eliminates geopolitical dependency on US AI. It cannot include export-controlled models anyway. The "sovereignty" framing is primarily a marketing positioning, not a genuine architectural independence.
Medium Severity
⏱️
Real-World Latency Concerns
Beta testers report Fugu-Ultra latency of ~108 seconds per query — impractical for interactive use cases. The paper reports Fugu ($7.32) is meaningfully cheaper than Claude Opus ($37.85) for comparable tasks, but beta tests found Fugu's output quality inferior in several real-world comparisons. The 11-benchmark evaluation may not reflect production workload distributions.
Medium Severity
🔁
Internal Inconsistency: Cheaper Beats Ultra
Fugu (smaller, faster, cheaper) outperforms Fugu-Ultra on 3 of 11 benchmarks: SciCode (60.1 vs 58.7), τ³ Banking (21.7 vs 20.6), Long Context Reasoning (74.7 vs 73.3). This undermines the "use Ultra for hard problems" positioning and suggests the additional orchestration overhead of Fugu-Ultra is not uniformly beneficial — domain-matching the coordinator variant is its own unsolved problem.
Low-Medium Severity
The Genuine Innovation vs. The Marketing Claim: The genuine technical contribution — that a small trained coordinator can adaptively route tasks across specialized frontier models and outperform any individual model — is real and well-supported. The surrounding narrative (frontier capability without export controls, parity with Fable 5/Mythos) is contested and partially inaccurate. Evaluate the architecture on its technical merits.
