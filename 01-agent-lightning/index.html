Why Does RL Exist?
Before diving into Agent Lightning, let's build intuition for why reinforcement learning exists вҖ” and why it's the only tool that works when you can't write the rules in advance.
Act 1 вҖ” The Problem
An LLM with tools... but no sense of order

Imagine someone messages the AI: "My flight just got cancelled. How do I get to Paris by tomorrow morning?"

The agent has three tools available. Stop and ask yourself вҖ” which one would you reach for first?

  • рҹ”Қ Search вҖ” look up alternative flights right now?
  • рҹ§  Memory вҖ” check the user's itinerary, budget, passport details?
  • рҹ§® Calculator вҖ” compare cost vs. travel time tradeoffs?

Search sounds right вҖ” but search for what? Without knowing where the user is flying from, their budget, or their flexibility, any search result is useless. The correct first step is Memory. An untrained agent doesn't know this.

The core problem: The right tool depends on context only the agent can discover step by step. There's no rule you can write in advance вҖ” and even a human expert pauses before answering.

Act 2 вҖ” The Solution
Learn from feedback, not from rules

Instead of programming the right answer, we give the agent a score after each attempt. Right answer = reward. Wrong answer = nothing.

Run training rounds and watch the agent figure out вҖ” through trial and error вҖ” that Memory must be called first, before searching for anything.

This is reinforcement learning in a nutshell: try вҶ’ score вҶ’ adjust вҶ’ repeat.

Key insight: Nobody told the agent "check Memory first." It discovered the right order on its own, purely from the reward signal вҖ” "did the user actually get to Paris?"

Act 3 вҖ” The Scale-Up
From one tool call to a full reasoning chain
Single Tool Call вҖ” One Decision

The agent makes one decision: which tool to call. RL handles this well вҖ” one action, one reward.

Simple. Standard RL handles this fine. One action вҶ’ one reward signal.

Multi-Step Chain вҖ” Complexity Explodes

A real agent task needs multiple tool calls in sequence. Each step is a decision. The final reward only comes at the end вҖ” so which step deserves credit?

The reward arrives at the end. But was it the Search step that was useful? The Calculator? Both? Standard RL can't tell вҖ” this is the credit assignment problem.

Agent Lightning вҖ” Credit Flows Back Through Every Step

Agent Lightning formalizes the entire chain as a structured MDP. The reward signal is decomposed and assigned back to each step that contributed вҖ” so every decision can be improved.

The purple reward arrows flow backwards through every step simultaneously. Agent Lightning is the first framework built to handle this at scale вҖ” for any tool, any agent architecture.

Agent Lightning is built specifically for the multi-step agent world вҖ” it knows which step in a long tool-calling chain deserves credit when the final answer is right.

Core Concept
What is a Structured MDP?
Agent Lightning says it "formalizes agent trajectories as a structured MDP." But what does that mean? Let's build this from the ground up вҖ” State, Action, Reward вҖ” then see what makes Agent Lightning's version "structured."
State (s) вҖ” What the agent knows

At every moment, the agent sees a state вҖ” its current snapshot of the world. It's everything the agent can observe right now.

For our cancelled-flight agent, the state at step 1 is just the user's message. At step 2, the state includes the user's message plus the Memory tool result. Each step the state grows richer.

Key point: The agent cannot go back in time. It only acts on the current state. So the state must contain everything relevant вҖ” which is why it accumulates tool results as the conversation progresses.

Action (a) вҖ” What the agent decides

Given the state, the agent picks an action вҖ” which tool to call, or what response to give. This is the decision the policy ПҖ learns to make.

An untrained agent might call рҹ”Қ Search immediately. A well-trained agent learns to call рҹ§  Memory first вҖ” because without knowing the user's origin city and budget, any search result is meaningless.

The policy ПҖ is just a function: given state s, output a probability distribution over actions. Training changes these probabilities вҖ” good actions get higher probability, bad actions lower.

Reward (r) вҖ” Feedback that shapes the policy

After each action, the agent gets a reward вҖ” a number that says "was that useful?" High reward вҶ’ do that more. Low reward вҶ’ do that less.

Over thousands of training iterations, the rewards shape the policy until the agent reliably picks the right action order вҖ” without anyone ever writing "check Memory first" as an explicit rule.

The credit problem: In a multi-step task, the final reward (+1 for correct answer) arrives at step 5 вҖ” but step 1 (calling Memory) might have been the most important decision. Standard RL doesn't know this. Agent Lightning's structured MDP fixes it.

The "Structured" Part

Now that you know S, A, R вҖ” here's what makes Agent Lightning's MDP structured. Toggle between the two approaches to see the difference:

reward only at end
Standard MDP: One reward arrives at the very end. All 3 intermediate steps share the same terminal signal вҖ” the agent can't tell which step actually mattered.

This is exactly what Agent Lightning's Unified Data Interface does: it takes any multi-turn agent trajectory and structures it into a list of (input, output, reward) transitions. Once structured, any RL algorithm вҖ” PPO, GRPO, REINFORCE++ вҖ” can train on it without needing to understand the agent internals at all.

Proximal Policy Optimization (PPO)
PPO fixes REINFORCE++'s instability problem by clipping the policy update. The agent can only change its behavior by a bounded amount per training step вҖ” preventing catastrophic forgetting.
L_CLIP(Оё) = E_t [ min( r_t(Оё)В·A_t, clip(r_t(Оё), 1-Оө, 1+Оө)В·A_t ) ] r_t(Оё) = ПҖ_Оё(a_t|s_t) / ПҖ_Оё_old(a_t|s_t) (how much the policy changed for this action)

Flight agent scenario: REINFORCE++ saw that Memory-first worked well and wants to make it far more likely. But if it swings the policy too hard, the agent might never try Search again вҖ” even when Memory returns empty. PPO clips the update: it can improve Memory-first, but only up to the Оө boundary per step.

PPO Objective вҖ” why the clip works

When A>0 (good action like Memory-first): PPO won't let the ratio r_t exceed 1+Оө, preventing over-reinforcement. When A<0 (bad action like Search-first): won't let ratio go below 1-Оө, preventing over-penalisation.

Why clipping matters for the flight agent

Without clipping, one lucky Memory-first trajectory could make the agent always call Memory regardless of context вҖ” even when the user hasn't provided their details yet. PPO ensures improvements are gradual and reversible.

In Agent Lightning

PPO is supported in Agent Lightning for stable training. Per-token advantage estimation extends naturally to LightningRL's transition-based format. Each flight-agent step is its own transition вҖ” PPO updates each one independently with its own clip.

GRPO вҖ” Group Relative Policy Optimization
GRPO's key insight: instead of a separate critic network to estimate "how good is this state?", just sample multiple trajectories from the same prompt and compare them against each other.
A_i = (r_i вҲ’ mean(r)) / std(r) (normalized relative to the group) L_GRPO = E_i [ min( r_i(Оё)В·A_i, clip(r_i(Оё),1вҲ’Оө,1+Оө)В·A_i ) ] вҲ’ ОІВ·KL(ПҖ_Оё вҖ– ПҖ_ref)

Flight agent scenario: Given the same prompt вҖ” "My flight got cancelled, how do I reach Paris?" вҖ” the agent generates 4 different trajectories in parallel. GRPO compares them against each other: trajectories above average get positive advantage (do more of this), below average get negative (do less). No critic needed вҖ” the group is the baseline.

4 Trajectory Attempts вҖ” same prompt, different strategies

Click any trajectory card to resample its reward and watch the normalized advantages update in real time.

Normalized Advantages A_i вҖ” who should be reinforced?

Positive advantage = above group average вҶ’ increase probability. Negative = below average вҶ’ decrease probability. The baseline is the group mean, not a separate neural network.

What happens after this GRPO step?

The policy shifts: strategies with positive advantage become more probable, strategies with negative advantage become less probable. After enough training rounds, Memory-first dominates вҖ” the agent has learned the right tool order without anyone writing a rule.

GRPO vs PPO
  • No value/critic network needed
  • Lower GPU memory footprint
  • Advantage estimated from group peers
  • Naturally handles open-ended generation
  • Ideal for LLM agents: each prompt gets its own group baseline

LightningRL vs. Masking Approaches
Most RL frameworks concatenate all turns and use token masking to ignore system prompts. Agent Lightning uses a fundamentally different approach that avoids the pitfalls of masking entirely.
Single-Turn RL вҖ” The Baseline

The simplest approach: one LLM call produces one training sample. Clean and straightforward, but ignores multi-step structure entirely.

Input prompt вҶ’ LLM call вҶ’ Output вҶ’ Reward One training sample per task.

Works well for basic QA and single-step tasks. But for tool-using agents that make 3вҖ“10 LLM calls per task, this throws away most of the signal вҖ” only the final call gets trained.

Masking вҖ” The Old Approach (Criticized by the Paper)

Concatenate all turns into one long sequence. Mask out system prompts, tool results, and context вҖ” only train on agent output tokens.

[SYS]Youareahelpfulassistant Think:checkMemoryfirst [TOOLRESULT]itinerary:CDG,budget:вӮ¬400 Think:nowsearchflights [TOOLRESULT]Eurostar7am,вӮ¬89 Answer:TakeEurostar.
A Agent tokens (trained) A Masked tokens (ignored)

Problems (from the paper):
вҖў Disrupts RoPE positional encodings вҖ” token positions become meaningless across masked gaps
вҖў Tight coupling with agent code вҖ” must modify agent internals to emit masks
вҖў Complex masks slow GPU kernels вҖ” irregular patterns prevent efficient batching
вҖў Hard to maintain across agent frameworks

LightningRL вҖ” The Agent Lightning Approach

Decompose the full trajectory into clean individual transitions. Each transition is a fresh (input, output, reward) tuple. No masking needed at all.

Trajectory вҶ’ [(inputвӮҒ, outputвӮҒ, rвӮҒ), (inputвӮӮ, outputвӮӮ, rвӮӮ), (inputвӮғ, outputвӮғ, rвӮғ)] Each transition = one self-contained training example.
Benefits
  • No masking вҖ” token positions always valid
  • Zero agent code modification needed
  • Works with any framework (LangChain, AutoGen, etc.)
  • Standard RL algorithms work unchanged
  • AIR provides intermediate rewards automatically
